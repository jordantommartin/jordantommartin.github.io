Dev Diary Week 7

July 10th, 2021

This week I worked on getting some majors bugs fixed in the system, adding objects to the simulation, and some further polish on previous features, like animations and the UI Watch Window.

Fixing Bugs: Always use Smart Pointers

One of the major goals for this week was to solve more bugs inside of the GOAP system. The main issue I was having was in the function BuildGraph() which is used to create all the possible plans that an agent could take at a certain time. I implemented this using a LinkedList style Node that carried all the information for a specific action and a pointer the the next node. Each of these linked lists would represent a plan and each plan would be placed into an array that was returned to the planner to decide which plan to take.

The issue was that the LinkedList would sometimes have garbage data, depending on the plans that it tried to create. I was able to figure out that the function was working fine, but when I returned from the function all the pointers were screwed up. At first I tried to change the way I was looping through the linked list to be index-based instead of a while loop, which did alleviate some of the issues. However, when I tried to add more actions to the plan, the issue came back.

After thinking about the fact that the pointers were being filled with garbage data once I returned from the function, I realized that the issue was that the data that the pointers were pointing to was allocated on the stack inside of a recursive function. This means this data would go out of scope and the pointer would be pointing to where the data used to live, but is now free. The solution was to use smart pointers to account for this situation. After refactoring my code to use smart pointers, the issue was resolved.

Adding Objects to the Simulation

After the issue with pointers was resolved, I started work on adding objects to the simulation. Originally during updates and presentations, I referred to these as "Smart Objects". This week I looked into exactly what the details and Smart Objects are much more complicated than I have time for. Essentially, the objects themselves act as Agents inside of the simulation and there has already been lots of research into this as a topic of itself. Due to this, I made an implementation of objects myself which means that actors can add any item in the world to their inventory and this item can be depleted later to do actions in the simulation.

To demonstrate this, I added a Vendor agent who can sell the Customer a ticket to get onto the ride. In order to get in line for the ride, the customer must buy a ticket from the vendor. The vendor loses one ticket from their inventory and the customer gains a ticket in their inventory. Once the customer is in line at the ride, the Ride Operator agent will come over and check if the Customer has a ticket. If the Customer does, that ticket is depleted and the customer can now go on the ride. You can see this in action in the final demo video.

UI and Animations

Lastly, I made some small changes to some previous features. First, I added so that the UI Watch Window will update with each Agent's name and that each completed action will light up green once it is completed. In addition to the UI, I also added animations to the actions, so now an agent can complete all of the three possible actions in the GOAP system (Move To, Animate, Object).

Dev Diary Week 5

June 26th, 2021

This week, I was able to solve some of core issues with my Unreal project that I struggled with last week and added some new features which help show off the simulation during next week's presentation.

Fix ALL the Bugs

During last week, I ran into several different issues with my Unreal project that impeded my progress. This week I was able solve these issues, which allowed me to begin making progress.

Bug Fix
When an agent was spawned, the AIController was never set. If it was placed in the level, the AIController worked fine.
  • Set the Auto Possess AI setting on the pawn to "Placed in World or Spawned"
Vanilla C++ classes in Unreal cannot use the delegate system to pass variables to a blueprint function. This was blocking me from passing actions from the planner C++ class to the UI Widget Blueprint.
  • Delegates can only communicate between blueprints and classes which uses the UCLASS macro. I created a helper function in my Action class which derives from the UActorComponent class to get around this issue.
STL containers in Unreal can sometimes have unpredicatable behavior when copying elements.
  • After doing some research, it is generally advised against using STL containers in Unreal C++ code. I refactored all my code to use TArrays, TMaps, TQueues.
Agents would never be able to reach their location because of their capsule colliders being too large.
  • On the agent C++ class, I created a custom trigger capsule that would detect if an agent collides with another agent. This condition also satisfies as reaching the exact destination.

Inventories

Once I had solved the issues listed above, I was able to complete adding inventories to each of the agents. Each agent will have its own inventory instance which allows them to add and remove AActor pointers and find any AActor pointer with a certain tag. With this inventory system, we can have items in the world that belong to certain agents. Then when executing a plan, the actions can access those inventories to check for certain items and quantities.

For example, I've set up to where once a Customer agent gets in line for a ride, it will wait until a free ride to become available. Once that ride is available, it then adds the ride to its inventory and it is no longer free to any of the other customers.

Game Elements

With those bugs aside, I was able to complete/add features to the Unreal project.

  • Added RTS style camera where you can select any agent by clicking on them and rotate/zoom
  • Speed buttons that allow you to change how fast the simulation is running
  • Watch window that shows the current plan of the agent that is selected

Below is a video that shows all of these elements in action.

Challenges

  • Collision on the agents required an additional custom capsule component.
  • The UI element still has many edge cases to work out before it can be used easily.
  • The Theme Park demo needs much more work before it is shown at presentation.

Moving to Unreal

For the majority of this week, I worked on taking the GOAP system that I started inside of Unity and moved that over to Unreal Engine. This included the 5 classes required for the base of the system: Agent, Action, Planner, World, and WorldStates. Being that it had to be written in C++, it required that I used different data structures and make use pointers, but otherwise the functionality is the same.

The theme for my project is a theme park, so for this test scene I created a Janitor agent and had him move to the Break Room. The Janitor has a goal of "isWaiting" and has an action of GoToBreakRoom which can accomplish that goal.

Unity Demo

Continuing with the Unity Demonstration to further the base GOAP system, next up was multi-step plans and multiple agents.

Creating the multi-step plan required creating a series of actions that satisfy the next action's preconditions with the current action's effects in order to reach a final state that satisfies the current goal. To accomplish this, I created several actions to be completed in succession in the hospital demo. The patient would have 3 actions that would create a plan which satisfies the goal of "isWaiting". These actions would be GoToHospital, Register, and GoToWaitingRoom. Once I attached all of those actions to the patient, the planner was able to take those actions and create a plan to get to the waiting room.

Once I had this complete, I moved onto adding a second agent into the plan. Following the hospital demo, I created a nurse that would attempt to treat the patient once they arrived at the waiting room. The nurse had a goal of treatPatient and had the action GetPatient which would satisfy that goal. However, this action was different as it now depends on another agent's actions in order to be executed. Before, all of the preconditions and effects were self contained within the agent's available actions and did not depend on the world itself.

To handle this, I added a post perform function to the patient's actions which would save a state of "Waiting" in the WorldStates singleton. The planner would then get that state from the WorldStates singleton and see that it can now satify the preconditions of the nurse's GetPatient action. Once this was set up, the patient would complete their multi-step plan and right afterward the nurse would go find the patient.

Dev Diary Week 2

June 2nd, 2021

One of the first things I worked on was researching other implementations of GOAP AI systems. I had found a Udemy lecture titled Advanced AI For Games with Goal-Oriented Action Planning that was very helpful in explaining the concepts in detail while providing a software design to get started with. This tutorial was taught using Unity and coded in C#.

Using this tutorial, I created 5 classes to start making the system:

  • Agent - Any of the NPCs that will be controlled using the GOAP system
      • Contains a lists of goals and further subgoals
      • Using the actions it has available, asks the planner a list of actions to accomplish the goal
      • If it has a list of actions, dequeues and performs each action
  • Action - Any of the actions that an agent use to achieve a goal
      • Contains a list of preconditions and after effects to be checked with the WorldStates
      • Determines if the action is achievable given the current WorldState
  • Planner - Builds a graph of possible plans and sorts it based on cost of actions
      • Uses a linked list style node to build graphs of actions
      • Given a list of possible actions, a goal to acheive, and the current WorldStates, builds a graph of possible plans
      • Sorts all of the possible plans based on costs and return a lists of actions for the chosen plan
  • World States - A class that maintains a map of pairs representing states of the world
      • Each state represents a fact within the world
      • Contains methods to check, modify, add, and remove states
  • World - A singleton that allows access to the map of WorldStates
      • Allows access to the WorldStates from any class

Once I had the beginnings of a GOAP system in place, I was able to create a simple one-step plan. In this hospital example, the patient was a given a goal of "isWaiting" meaning that he would want to go to the waiting room. He was then given an action called "GoToHospital" with a target of the Waiting Area in the game. It had a cost of one with no preconditions. It had one after effect that would add a state of <"isWaiting", 1> to the map, indicating that it is true that the patient is now waiting.

With this goal and action given to the agent, I was able to execute the simple one-step plan below! From here, I will continue to watch this tutorial as it also explains how to create multi-step plans.

To start off, I got familiar with Unreal's NavMesh by moving some agents around. I created a C++ class for an NPC and created a function to find a random point on the NavMesh and then input this into the MoveTo function on the AIController.

From here, I worked on setting up C++ classes using the same software design as the tutorial mentioned eariler. This is currently in progress, as I am working on setting this up in C++ using different methods.

What is Goal Oriented Action Planning?

Goal Oriented Action Planning is a technique of artificial intelligence used to control agents (NPCs) in a game. This is opposed to some more traditional techniques such as Finite State Machines and Behavior Trees. It was based on a technology called Automated Planning that has been used outside of video games for some time. It was popularized in video games in the mid-2000s when the game F.E.A.R. utilized this system to control its NPCs. Since then, it has been included in many other AAA games and evolved into even more complicated AI techniques.

How does Goal Oriented Action Planning Work?

Goal Oriented Action Planning works by attempting to figure out a sequence of actions that will achieve a distant "goal" for an agent. It does this by querying the World State for facts and creating a plan from a set of given actions. These actions are specifically made up of three things:

  • Objects - Any items in the world involved in the action
  • Preconditions - Facts that must be true for the action to work
  • Effects - How the world state changes when the action is complete

Why use Goal Oriented Action Planning?

Goal Oriented Action Planning has some advantages and disadvantages that can be considered when thinking of using it inside a game.

Pros

  • Actions are decoupled from agents
  • Smaller, more data-driven approach to AI
  • Allows for priority of multiple goals at one time

Cons

  • Revalidation of plan during execution
  • The planner makes all decisions for the agents
  • Performance overhead if agents are not in the scene

The GOAP State Machine, which is much simpler that a typical finite state machine.

Rewinding Destructible Meshes in Unreal Engine

January 8th, 2021

Over the winter break, I worked on prototyping a feature in which you can rewind the destruction of an ancient shrine. The goal was to use a destructible mesh so that we could save our artists the time it would take to do this manually through animation.

At first we looked into using the Chaos Destruction system that is in beta at the moment. However, after having to do a source build of the engine and encountering a few bugs, we decided that this wouldn't be the best route. I then had a team member suggest using the APEX Destruction plugin that allows you to create destructible meshes, which worked much better.

Rewinding Mechanic

Using a video tutorial provided by Crystal Clear Game Studios on Youtube, I was able to create a rewind mechanic that can be applied to any actor. Essentially, this involves saving off the transform and velocity of the object each frame to an array. When rewinding, you set the transform and velocity to last element and pop it off the array.

Destructible Meshes

After setting up the rewind mechanic, I attempted to apply the same logic to destructible meshes. For simplicity, I used a cube with 3 pieces. After researching and working with destructible meshes, I found that they operate similar to skeletal meshes and even have bones for each piece of the destruction. UE4 allows you to GET the transform of these bones but doesn't allow you to SET the transform.

I did find that you can set the transform of a bone in a Poseable Mesh, however. This would work if we could use the physics of the destructible mesh when destroying an object and using the poseable mesh when rewinding the transforms. Then use a visibility toggle on each of the components depending on whether we are playing or rewinding.

So, I attempted to convert the destructible mesh into a poseable mesh. I exported the destructible mesh as an FBX and then reimported it into UE4 as a skeletal mesh. Then I added a poseable mesh component to the blueprint, which used this skeletal mesh. Bingo, I was able to manipulate the bones by setting the transform! However, it didn't look quite right. Each of the pieces was moving correctly, but the original shape was also being manipulated. So I needed to remove the original cube shape from the poseable mesh, so that only the pieces are being manipulated.

After fiddling around in Maya, I realized I wasn't just going to be able to delete the original shape, as that just ruined the entire skeleton. Instead, I applied a transparent material to the original shape. Success! Only the pieces are shown inside of UE4.

So, after creating and modifying an identical poseable mesh, I was able to apply the rewind mechanic to each of the bones. Then use the destructible mesh for playing forward and the poseable mesh for rewinding backward.
